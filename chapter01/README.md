## 一、信息就是位+上下文

### 核心概念

系统中的所有信息——包括磁盘文件、内存中的程序、内存中存放的用户数据以及网络上传送的数据，都是由一串bit（位）表示的。区分不同数据对象的唯一方法是我们在什么\*\*上下文（context）\*\*中看待这些数据。

> 示例：[hello.c](hello.c) 中 `hello` 字符串在不同上下文中的表示：
>
> - 源程序中：ASCII字符 `h` `e` `l` `l` `o`，分别对应十进制 104 101 108 108 111
> - 底层存储：每个字符占1字节，共5字节
> - 整数上下文：同样的bit模式可以被解释为一个数值

### 关键要点

- **位模式相同，含义取决于上下文**——同样的比特流在不同上下文中被解释为不同的数据类型
- 源程序本质上是一个由值0和1组成的位序列（字节序列）
- 大部分现代系统使用ASCII标准表示文本字符
- 只由ASCII字符构成的文件称为**文本文件**，其他文件称为**二进制文件**

## 二、程序被其他程序翻译成不同格式

### C程序编译过程：预处理 → 编译 → 汇编 → 链接

```
hello.c (源程序/文本文件)
   │
   ▼  预处理器 (cpp)
hello.i (修改了的源程序/文本文件)
   │
   ▼  编译器 (ccl)
hello.s (汇编程序/文本文件)
   │
   ▼  汇编器 (as)
hello.o (可重定位目标程序/二进制文件)
   │
   ▼  链接器 (ld)
hello (可执行目标程序/二进制文件)
```

### 各阶段详解

| 阶段      | 工具        | 输入                 | 输出      | 说明                                                       |
| ------- | --------- | ------------------ | ------- | -------------------------------------------------------- |
| **预处理** | 预处理器(cpp) | hello.c            | hello.i | 处理`#include`等预处理指令，将头文件内容直接插入程序文本                        |
| **编译**  | 编译器(ccl)  | hello.i            | hello.s | 将C语言程序翻译为汇编语言程序（文本格式）                                    |
| **汇编**  | 汇编器(as)   | hello.s            | hello.o | 将汇编语言翻译为机器语言，打包成**可重定位目标程序**(relocatable object program) |
| **链接**  | 链接器(ld)   | hello.o + printf.o | hello   | 合并多个可重定位目标文件（如标准C库中`printf`对应的目标模块），生成**可执行目标文件**        |

### 为什么要理解编译系统？

1. **优化程序性能**：理解编译过程才能写出高效代码
2. **理解链接时错误**：解决链接阶段的符号解析、静态库与动态库问题
3. **避免安全漏洞**：理解缓冲区溢出等安全问题的根源

## 三、处理器读并解释存储在内存中的指令

```

                    │              系统总线 (System Bus)        │
                    └─────────────────────────────────────────┘

                    │ ├───────┤ │              │
                    │ │寄存器堆│ │              │
                    │ ├───────┤ │              │

                    └───────────┘              │
                                               │
                    ┌───────────┐              │

                    └─────┬─────┘
                     ┌────┴────┐
               ┌─────┴──┐  ┌───┴──────┐
               │存储器总线│  │ I/O总线   │

               │ 0xFFFF │  │ │键盘  │ │
               └────────┘  │ └──────┘ │
                           └──────────┘
```

### 核心硬件组件

#### 1. 总线 (Buses)

- 贯穿整个系统的电子管道，携带信息字节并负责在各个部件间传递
- 通常被设计成传送定长的**字(word)**，字长是系统的一个基本参数（32位系统字长4字节，64位系统字长8字节）

#### 2. I/O设备 (I/O Devices)

- 系统与外部世界的联系通道
- 每个I/O设备通过\*\*控制器(controller)**或**适配器(adapter)\*\*与I/O总线相连
  - **控制器**：I/O设备本身或系统主板上的芯片组
  - **适配器**：插在主板插槽上的卡
- 用户输入示例：键盘 → USB控制器 → I/O总线 → I/O桥 → 系统总线 → 主存

#### 3. 主存 (Main Memory)

- 临时存放程序和处理程序所需的数据
- 物理上：由\*\*DRAM（动态随机存取存储器）\*\*芯片组成
- 逻辑上：线性的字节数组，每个字节有唯一地址（从0开始）

#### 4. 处理器 (Processor/CPU)

- **程序计数器(PC)**：指向主存中某条机器语言指令（大小为一个字）
- 处理器不断执行PC指向的指令，再更新PC指向下一条指令
- 指令执行模型（简化）：
  - **加载(Load)**：从主存复制一个字节或字到寄存器，覆盖原内容
  - **存储(Store)**：从寄存器复制一个字节或字到主存某个位置，覆盖原内容
  - **操作(Operate)**：把两个寄存器的内容复制到ALU做算术运算，结果存回寄存器
  - **跳转(Jump)**：从指令中抽取一个字复制到PC，覆盖PC原值

### hello程序的执行过程

1. **Shell读取命令**：用户输入 `./hello`，Shell（命令行解释器）程序将字符逐一读入寄存器，再存入主存
   > **为什么经过寄存器？** 这是\*\*程序化I/O(Programmed I/O)\*\*方式：I/O设备本身不能直接访问主存，操作系统内核通过CPU执行`in`指令从I/O设备读取数据到寄存器，再执行`mov`指令写入主存。相比之下，**DMA**（见步骤2）允许I/O设备在DMA控制器协助下直接访问主存，无需CPU介入。
2. **加载程序**：Shell调用`execve`系统调用，通过DMA（直接存储器存取）**不经过处理器**直接将hello可执行文件的代码和数据从磁盘复制到主存
3. **执行指令**：处理器开始执行hello程序的机器语言指令
4. **输出结果**：将 `hello world\n` 字符串从主存复制到寄存器，再复制到显示设备

> **关键洞察**：程序执行过程中，大部分时间花在**复制数据**上——在不同硬件组件之间移动数据

## 四、高速缓存至关重要

### 为什么需要缓存？

处理器与主存之间的速度差距巨大——处理器读主存需要多个周期，而寄存器访问几乎瞬间完成。这种**处理器-存储器差距**是计算机系统面临的主要挑战之一。

### 高速缓存存储器 (Cache Memory)

```
┌──────────┐    L1缓存     ┌──────────┐    L2缓存     ┌──────────┐    L3缓存     ┌──────────┐
│  CPU寄存器 │◄────────────►│  L1高速缓存│◄────────────►│  L2高速缓存│◄────────────►│  L3高速缓存│◄──► 主存
│ (数百字节)  │  ~1ns       │ (数十KB)  │  ~几ns       │ (数百KB)  │  ~十几ns      │ (数MB)    │     (GB)
└──────────┘              └──────────┘              └──────────┘              └──────────┘
```

| 缓存层级 | 典型大小            | 典型访问时间           | 由什么实现 |
| ---- | --------------- | ---------------- | ----- |
| L1   | 数十KB（如32KB-64KB） | \~1ns (4个时钟周期)   | SRAM  |
| L2   | 数百KB（如256KB）    | \~几ns (10个时钟周期)  | SRAM  |
| L3   | 数MB             | \~十几ns (40个时钟周期) | SRAM  |
| 主存   | 数GB             | \~50-100ns       | DRAM  |

- **L1缓存**：位于CPU芯片上，几乎和寄存器一样快
- **L2缓存**：同样位于CPU芯片上，比L1稍慢但容量更大
- **L3缓存**：多核共享，比L2更慢但仍远快于主存
- 利用**局部性原理(locality)**：
  - **时间局部性(temporal locality)**：最近访问过的数据很可能在近期再次被访问
  - **空间局部性(spatial locality)**：最近访问过的数据附近的数据很可能在近期被访问

### 高速缓存的意义

- 利用高速缓存可以将程序性能提升一个数量级甚至更多
- 理解缓存原理是写出高效程序的关键

## 五、存储设备形成层次结构

### 存储层次结构图

```
  ┌─────────────┐  更快、更贵、更小
  │  L0: 寄存器   │  ← CPU内部
  ├─────────────┤
  │  L1: L1缓存   │  ← SRAM (On-Chip)
  ├─────────────┤
  │  L2: L2缓存   │  ← SRAM (On-Chip)
  ├─────────────┤
  │  L3: L3缓存   │  ← SRAM (共享)
  ├─────────────┤
  │  L4: 主存     │  ← DRAM
  ├─────────────┤
  │  L5: 本地磁盘  │  ← 机械硬盘/SSD
  ├─────────────┤
  │  L6: 远程存储  │  ← 网络文件系统/分布式存储
  └─────────────┘  更慢、更便宜、更大
```

### 核心思想

- **上一层是下一层的缓存**：L1是L2的缓存，L2是L3的缓存……主存是磁盘的缓存
- 每一层之间通过缓存机制传递数据
- 程序员可以利用局部性原理，使程序主要访问层次结构上较高的层，从而获得更好性能

## 六、操作系统管理硬件

### 操作系统的两个基本功能

1. **防止硬件被失控程序滥用**——保护
2. **向应用程序提供简单一致的机制来控制复杂且大不相同的硬件**——抽象

### 操作系统的三大抽象

```
  ┌─────────────────────────────────────────────┐
  │              应用程序                         │
  ├─────────────┬───────────────┬───────────────┤
  │   进程       │    虚拟内存     │     文件       │  ← 操作系统抽象
  ├─────────────┼───────────────┼───────────────┤
  │  处理器+主存  │     主存       │    I/O设备     │  ← 硬件
  └─────────────┴───────────────┴───────────────┘
```

#### 1. 进程 (Process)

- 进程是对处理器、主存和I/O设备的抽象
- 操作系统通过进程给每个程序三种关键错觉：
  - 独占处理器（通过上下文切换，让多个进程轮流使用CPU）
  - 独占主存（通过虚拟内存，让每个进程拥有独立地址空间）
  - 独占I/O设备（通过文件抽象，隐藏其他进程和系统细节）
- 操作系统通过\*\*上下文切换(context switch)\*\*实现多进程并发执行
  - 保存当前进程的上下文（PC、寄存器等CPU状态）
  - 恢复新进程的上下文
  - 将控制权传递给新进程
- **进程与线程**：一个进程可以由多个称为\*\*线程(thread)\*\*的执行单元组成，线程运行在进程的上下文中，共享同样的代码和全局数据。线程比进程更轻量，是并发编程的重要工具

#### 2. 虚拟内存 (Virtual Memory)

- 虚拟内存是对主存的抽象，使用磁盘作为后备存储
- 虚拟内存给每个进程一种**错觉**：仿佛自己独占了主存
- 每个进程看到的都是一致的私有地址空间
- 虚拟内存将地址空间划分为若干**页(page)**，每页大小固定，在需要时才从磁盘换入主存

**Linux进程的虚拟地址空间结构**（以32位系统为例，从低地址到高地址）：

```
  ┌──────────────────────┐ 0xFFFFFFFF (32位) / 0x7FFFFFFFFFFF (64位用户空间顶部)
  │     内核虚拟内存       │  ← 用户代码不可见，必须通过系统调用访问
  │  (内核代码和数据)      │
  ├──────────────────────┤ ─── 用户空间与内核空间分界 ───
  │     用户栈 ↓          │  ← 运行时创建，向下(低地址)增长
  │                      │
  ├──────────────────────┤
  │     共享库的内存映射区域 │  ← 如C标准库、printf等共享库代码
  ├──────────────────────┤
  │                      │
  │     运行时堆 ↑        │  ← malloc分配，向上(高地址)增长
  ├──────────────────────┤
  │     未初始化数据段(.bss)│  ← 未初始化的全局/静态变量
  ├──────────────────────┤
  │     已初始化数据段(.data)│ ← 已初始化的全局/静态变量
  ├──────────────────────┤
  │     只读代码和数据段    │  ← 可执行目标代码(.text)、只读数据(.rodata)
  └──────────────────────┘ 0x00000000 (低地址)
```

| 区域                 | 说明                        |
| ------------------ | ------------------------- |
| **内核虚拟内存**         | 操作系统内核代码和数据，用户进程不可直接访问    |
| **用户栈**            | 函数调用、局部变量存储，向低地址增长        |
| **共享库映射区**         | 共享库(如libc)的代码和数据          |
| **堆**              | 动态分配内存(malloc/new)，向高地址增长 |
| **未初始化数据段(.bss)**  | 未初始化的全局和静态变量，运行时初始化为0     |
| **已初始化数据段(.data)** | 已初始化的全局和静态变量              |
| **代码段(.text)**     | 可执行机器指令和只读数据(.rodata)     |

#### 3. 文件 (File)

- 文件是对I/O设备的抽象
- Unix中**一切皆文件**：所有I/O设备（包括键盘、显示器、磁盘、网络）都被模型化为文件
- 统一的文件操作接口：`open()`, `read()`, `write()`, `close()`
- 本地磁盘可以看做远程文件系统的高速缓存

## 八、重要主题

### 1. Amdahl定律

**核心公式**：
$$S = \frac{1}{(1-\alpha) + \alpha/k}$$

- $\alpha$：某部分**原始执行时间**占系统总原始执行时间的比例
- $k$：该部分性能提升的倍数
- $S$：整体系统加速比

**关键结论**：

- 要想显著加速整个系统，必须提升系统中**大部分**的性能
- 即使大幅改进系统某一部分，如果该部分只占总时间的一小部分，整体加速效果也有限
- **例**：某部分占60%时间，加速3倍 → 整体加速比 = 1/(0.4+0.6/3) = 1/0.6 ≈ 1.67倍

### 2. 并发与并行

| 层次                 | 类型      | 说明                 |
| ------------------ | ------- | ------------------ |
| **线程级并发**          | 多进程/多线程 | 伪并发（单核时间片）或真并发（多核） |
| **指令级并行**          | 流水线/超标量 | 处理器同时执行多条指令        |
| **单指令多数据并行(SIMD)** | 向量指令    | 一条指令并行处理多个数据       |

- **多核处理器**：将多个CPU集成到一个芯片上，每个核有自己的L1和L2缓存，共享L3缓存
- **超线程(HT)**：Intel的超线程技术是SMT（同时多线程）的一种实现，允许单个处理器核维护两套独立的架构状态（寄存器组、PC等），从而在操作系统看来如同两个逻辑核。当一个线程因缓存未命中等原因停顿时，硬件可快速切换到另一个线程继续执行，提高处理器资源的利用率

### 3. 抽象的重要性

计算机系统中的核心抽象：

| 抽象             | 底层实现          |
| -------------- | ------------- |
| **指令集架构(ISA)** | 实际处理器硬件       |
| **进程**         | 处理器、主存、I/O设备  |
| **虚拟内存**       | 主存（磁盘作后备存储）   |
| **文件**         | I/O设备         |
| **虚拟机**        | 整个计算机（包括操作系统） |

> 指令集结构提供了对实际处理器硬件的抽象，使用这个抽象，机器代码程序员可视自己的程序运行在一个一次只执行一条指令的处理器上。而底层的硬件比抽象的指令集要复杂精细的多，他并行执行多条指令，但是总是与那个简单有序的模型保持一致。

## 十、利用hello.c的运行过程漫游

[hello.c](hello.c)

hello程序从源文件到最终在屏幕上显示"hello world"，经历了：

1. **创建源文件**：程序员编写hello.c
2. **编译**：预处理→编译→汇编→链接，生成可执行文件
3. **加载**：Shell通过系统调用将可执行文件从磁盘加载到主存
4. **执行**：CPU逐条执行机器指令
5. **输出**：数据从主存→寄存器→显示设备

整个过程体现了**数据在不同组件之间的通信复制过程**，以及**缓存的重要性**——如果没有缓存，处理器将大量时间浪费在等待数据传输上。

## 十一、本章总结

计算机系统由**硬件**和**系统软件**组成，它们共同协作以运行应用程序。贯穿本章的核心思想包括：

1. **信息就是位+上下文**：系统中所有信息都是 bit 串，数据的含义由它被解释的上下文决定。
2. **程序被翻译执行**：C 程序通过预处理、编译、汇编、链接四个阶段变成可执行文件，最终被处理器执行。
3. **数据移动决定性能**：程序运行时，大量时间花在 CPU、主存、I/O 设备之间的数据复制上。
4. **缓存至关重要**：利用局部性原理，高速缓存缓解了处理器与主存之间的速度差距。
5. **存储设备形成层次**：从寄存器到远程存储，每一层都是下一层的高速缓存。
6. **操作系统通过抽象管理硬件**：进程、虚拟内存、文件三个核心抽象隐藏了硬件复杂性。
7. **抽象是计算机系统的基础**：从指令集架构到虚拟机，每一层抽象都使上层无需关心底层细节。
